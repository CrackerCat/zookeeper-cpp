1 # ZooKeeper C++
2 
3 A ZooKeeper client for C++.
4 
5 Features include (but are not necessarily limited to):
6 
7 - Simple
8  - Connect with just a connection string
9  - Clients should not require factories
10  - Does not require *any* knowledge of the Java or C APIs
11 
12 - Configurable
13  - Use the parts you need
14  - Change parts to fit in your application
15 
16 - Safe
17  - In the best case, illegal code should fail to compile
18  - An illegal action should throw an exception
19  - Utility functions have a [strong exception guarantee](http://www.gotw.ca/gotw/082.htm>)
20 
21 - Stable
22  - Worry less about upgrading -- the API and ABI will not change out from under you
23 
24 - Documented
25  - Consumable by human beings
26  - Answers questions you might actually ask

33 ## Usage
34 
35 Ultimately, the usage looks like this (assuming you have a ZooKeeper server running on your local host):
36 
37  #include <zk/client.hpp>
38  #include <zk/multi.hpp>
39 
40  #include <exception>
41  #include <iostream>
42 
44  template <typename T>
45  void print_thing(const std::future<T>& result)
46  {
47  try
48  {
49  // Unwrap the future value, which will not block (based on usage), but could throw.
50  T value(result.get());
51  std::cerr << value << std::endl;
52  }
53  catch (const std::exception& ex)
54  {
55  // Error "handling"
56  std::cerr << "Exception: " << ex.what() << std::endl;
57  }
58  }
59 
60  int main()
61  {
62  // zk::client::create returns a future<zk::client>, which is delivered when the connection is established.
63  auto client = zk::client::create("zk://127.0.0.1:2181")
64  .get();
65 
66  // get_result has a zk::buffer and zk::stat.
67  client.get("/foo/bar")
68  .then(print_thing<zk::get_result>);
69 
70  // get_children_result has a std::vector<std::string> for the path names and zk::stat for the parent stat.
71  client.get_children("/foo")
72  .then(print_thing<zk::get_children_result>);
73 
74  // set_result has a zk::stat for the modified ZNode.
75  client.set("/foo/bar", "some data")
76  .then(print_thing<zk::set_result>);
77 
78  // More explicit: client.create("/foo/baz", "more data", zk::acls::open_unsafe(), zk::create_mode::normal);
79  client.create("/foo/baz", "more data")
80  .then(print_thing<zk::create_result>);
81 
82  client.get("/foo/bar")
83  .then([client] (const auto& get_res)
84  {
85  zk::version foo_bar_version = get_res.get().stat().data_version;
86 
87  zk::multi_op txn =
88  {
89  zk::op::check("/foo", zk::version::any()),
90  zk::op::check("/foo/baz", foo_bar_version),
91  zk::op::create("/foo/bap", "hi", nullopt, zk::create_mode::sequential),
92  zk::op::erase("/foo/bzr"),
93  };
94 
95  // multi_res's type is zk::future<zk::multi_result>
96  client.commit(txn).then(print_thing<zk::multi_result>);
97  });
98 
99  // This is not strictly needed -- a client falling out of scope will auto-trigger close
100  client.close();
101  }

146 ### `zk/fake`
147 
148 This library also provides a fake version of ZooKeeper which operates in-memory.
149 It is meant to be used in your unit testing, when fine-grained control of behavior of ZooKeeper is needed.
150 This allows for the injection of arbitrary behavior into ZK, allowing you to simulate some of the hard-to-reproduce
151 issues like `zk::event_type::not_watching`, `zk::marshalling_error`, or timing bugs.
152 It also allows for fast creation and teardown of entire databases, which is commonly done in unit testing.
153 
154 It is connected to through using a connection string of the form:
155 
156  fake://{name}
157 
158 To use this in unit tests link to `libzkpp_fake` and use `zk::fake::server`:
159 
160  TEST(my_test)
161  {
162  // The default constructor uses a randomly-generated unique name
163  zk::fake::server server;
164 
165  // Fetch that name through the connection_string
166  zk::client client(server.connection_string());
167 
168  // use client normally
169  }

176 ## Unsupported Functionality
177 
178 If you are used to using ZooKeeper via the Java or C APIs, there are a few things that are explicitly not supported in
179 this library.
180 
181 ### Global Watches
182 
183 There are two main ways to receive watch notifications: the global watch or through use a watcher objects.
184 In the Java API, the `ZooKeeper` client allows for a global
185 [Watcher](https://zookeeper.apache.org/doc/r3.4.10/api/org/apache/zookeeper/Watcher.html).
186 In the C API, `zookeeper_init` can be provided with a global function with the signature
187 `void (*)(zhandle_t* zh, int type, int state, const char* path, void* watcherCtx)` to achieve this same result.
188 The ZooKeeper community considers global watches as "legacy" and prefers the use of watcher objects set on a per-path
189 basis.
190 As such, global watches are *not* supported by this library.
191 
192 ### Synchronous API
193 
194 The C library offers both a synchronous and an asynchronous API.
195 This library offers only an asynchronous version.
196 If you prefer a synchronous API, call `get()` on the returned `future` to block until you receive the response.
197 
198 ### Non-Linux
199 
200 Can you get this library working on platforms that are not Linux?
201 Maybe.
202 But Linux is the primary development, testing, and deployment platform of people writing distributed applications, so
203 this library is targetted at Linux.

215 ## F.A.Q.
216 
217 ### Why `erase` instead of `delete`?
218 
219 In the Java and C APIs, the act of removing a ZNode is called `delete` and `zoo_delete`, respectively.
220 However, `delete` is a C++ keyword and cannot be used as a member function.
221 So, this library uses `erase`, which falls in line with standard C++ containers.
222 Alternatives such as calling the operation `delete_` look a bit worse (in the author's opinion).
223 
224 ### Why are watch calls separate?
225 
226 In the Java and C APIs, adding a watch to a ZNode is an additional parameter to the `get`, `get_children`, or `exists`
227 calls while this library uses separate `watch`, `watch_children`, and `watch_exists` calls.
228 This is done because the return types are different between a simple fetch and setting a watch.
229 While `get` returns a `future<get_result>`, `watch` returns the slightly more complicated `future<watch_result>`.
230 The `future` in `watch_result::next()` would be disabled in cases where a flag is not set, and it would be ignored with
231 the majority of use cases.
232 This leads to an awkward API for simple calls.
233 
234 An alternative used by other libraries is to provide a `std::function`, implying to not watch when the function is not
235 passed in.
236 This has a number of disadvantages:
237 
238 - There is no good way to cancel a watch without giving an extra parameter.
239  With a `future`, you simply let it fall out of scope.
240 - Watches are delivered only once, which is obvious from a `future`-like API, but not obvious from a `function`-like
241  API.
242 - It is not obvious what the behavior should be if the original call returns in error.
243  With a `future`, the behavior is obvious, since you never receive the mechanisms to perform the watch.
244 
245 In Java, the method of choice is to use the
246 [Watcher](https://zookeeper.apache.org/doc/r3.4.10/api/org/apache/zookeeper/Watcher.html) interface, but this feels
247 extremely out of place in C++ code.
